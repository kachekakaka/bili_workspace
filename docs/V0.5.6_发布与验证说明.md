# bili_workspace V0.5.6 发布与验证说明

> 本文件是 V0.5.6 的历史发布记录。当前 `main` 已继续增加搜索模式、标签、删除历史、任务批量操作和新的 `userdata` 目录边界。日常部署和验收请同时阅读 `docs/README.md`、`docs/需求落实清单.md` 和 `docs/V0.5功能与验收.md`。

## 1. V0.5.6 目标

V0.5.6 把 Windows x64 运行环境和 Docker 交付链路整合进 GitHub 仓库，使正常使用流程变为：

```text
git pull
start.bat
```

Windows 不再要求预装 Python，也不在启动时访问 PyPI 或 GitHub Release。QNAP 默认拉取 GHCR 的 amd64/arm64 预构建镜像，同时保留本地构建后备。

## 2. Windows 集成运行时

GitHub Windows Runner 从固定来源构建并校验：

- Python 3.13.14 官方 64 位 embeddable package；
- `requirements.lock` 中固定版本的 Python 依赖；
- BBDown 1.6.3 官方 Windows x64 资产；
- 固定 `imageio-ffmpeg` Windows x64 wheel 中的 FFmpeg。

产物为：

```text
vendor/windows/python-runtime.pack
vendor/windows/media-runtime.pack
vendor/windows/runtime-manifest.json
```

两个 `.pack` 均为 ZIP 格式的普通 Git 二进制对象，每个文件小于 100 MiB，因此不依赖 Git LFS。构建时校验上游 SHA-256，包内生成逐文件清单，提交前执行 Python 导入、BBDown 与 FFmpeg 冒烟测试。

Windows 启动脚本会校验仓库清单、包 SHA-256、包内路径和逐文件哈希，再解压到被 Git 忽略的本地运行目录。实际配置、数据库、Cookie、媒体和缓存不会进入 Git。

## 3. Docker / QNAP

`compose.yaml` 默认使用：

```text
ghcr.io/kachekakaka/bili_workspace:latest
```

`.github/workflows/docker-image.yml` 构建并发布：

```text
linux/amd64
linux/arm64
```

需要在 NAS 本地构建时，在 `docker/.env` 设置：

```env
BUILD_LOCAL=true
```

## 4. 当前主线持久化目录

V0.5.6 发布后，主线把可变数据进一步整理为三类边界。当前部署必须使用：

```text
Windows
config/
userdata/
downloads/

Docker
/data/config
/data/userdata
/downloads
```

Docker 宿主机通过：

```text
CONFIG_DIR
USERDATA_DIR
MEDIA_DIR
```

映射。旧文档中的 `/data/media`、`/data/cache`、`/data/tmp` 四目录结构已经过时。

## 5. V0.5.6 后续主线变化

当前 `main` 在 V0.5.6 基础上增加：

- 原始、精准和模糊搜索；精准/模糊只对原始返回标题二次过滤；
- 搜索默认 10 个页码和“再显示10页”；
- 默认屏蔽已下载和已删除；
- 标签、无标签筛选、标签/分组 chip；
- 作品直接修改分组；
- 作品库按时间、标题、时长、大小、分组和标签升降序排序；
- 任务原 ID 重试、编辑画质和批量操作；
- 独立 `deleted_media` 删除历史；
- 搜索和创建下载后不自动跳转任务中心。

这些功能不改变 V0.5.6 的 Windows 便携运行时交付方式。

## 6. GitHub 仓库设置

运行时构建工作流需要仓库允许 GitHub Actions 写入内容：

```text
Settings → Actions → General → Workflow permissions → Read and write permissions
```

Docker 镜像第一次发布后，若希望 QNAP 无需登录即可拉取，还需将 GHCR 包可见性设为 Public；否则在 QNAP 上执行 `docker login ghcr.io`。

## 7. 验证标准

源码提交必须通过：

```text
tools/verify_source.py
python -m compileall
Ruff
pytest
前端 JavaScript 语法检查
Shell / Compose 静态检查
```

运行时工作流还必须通过：

```text
固定来源 SHA-256
安全解压
单文件小于 100 MiB
包内逐文件哈希
Portable Python 依赖导入
BBDown --help
FFmpeg -version
```

目标 Windows 最终验收：

```text
git pull --ff-only origin main
start.bat
网页可打开
真实搜索、清晰度预览、下载、混流和播放正常
```

当前功能验收请使用《V0.5.x 功能与验收清单》，不要只按本历史发布报告判断。
