# 任务所有权与保留策略

本文件描述当前 `main` 已实现的 V0.6 任务所有权、保留策略和对应多用户任务界面。

## 所有权来源

每个正式任务记录包含 `owner_user_id`。创建任务时，拥有者只能来自当前认证会话；客户端提交的 `owner_user_id`、`user_id`、角色或权限字段不会改变归属。管理员重试普通用户任务时，任务仍归原用户。

已有数据库升级到 schema v4 时，旧任务快照和设备导出统一归属原管理员。v3→v4 迁移不会再次撤销 PR 2 已创建的有效会话。

## 普通用户下载边界

普通用户提交下载时，后端固定覆盖为：

```text
destination = device
group_id = ""
group = ""
force = false
owner_user_id = 当前会话用户
```

因此手工构造媒体库目标、分组或他人拥有者也不能写入管理员媒体库。

## 权限隔离

普通用户只能访问自己的：

- 任务列表与详情；
- 任务日志和日志下载；
- 暂停、继续、取消、重试和删除操作；
- SSE 任务事件；
- 导出准备、文件下载和放弃导出。

访问其他用户的任务或导出统一按不存在返回，避免泄露对象是否存在。管理员可以查看和操作全部用户任务，并按用户、状态、目标、关键词、创建时间和完成时间查询。

## 任务界面

普通用户任务页只显示当前账号的任务和设备导出，不提供用户或媒体库目标筛选。管理员任务中心可按用户、状态、目标、关键词、排序字段和方向查询，也可按用户分组；每张管理员任务卡显示 `显示名（username）`。筛选和排序参数发送到 SQLite 查询，前端不先读取全部 JSON 再全量排序。

## 同作品设备导出

有效导出判定使用 `owner_user_id + source_key`。不同用户可以同时导出同一个 BV；设备导出队列的内部索引和临时产物按用户命名空间隔离，而 API 仍显示原始 BV/source key。

## 数量和保留规则

权威常量位于 `app/constants.py`：

```text
NORMAL_USER_ACTIVE_TASK_LIMIT = 10
NORMAL_USER_TASK_RETENTION_DAYS = 7
NORMAL_USER_TASK_HISTORY_LIMIT = 100
ADMIN_TASK_HISTORY_LIMIT = 500
```

普通用户 `queued + running` 达到 10 个后拒绝新任务。终态任务满足以下任一条件即清理：完成超过 7 天，或该用户终态历史超过 100 条。超过数量时按完成时间最旧优先。

管理员终态任务最多保留 500 条，不应用普通用户 7 天规则。活动任务、正在取消的任务和仍处于准备/可下载/清理等待状态的导出不会被历史清理。

清理同步删除任务记录、兼容快照、任务日志、失效导出元数据及对应普通下载审计记录；临时文件仍遵守设备导出 TTL 和完整传输后立即删除规则。

## 迁移与回滚

升级前应用使用 SQLite `backup()` 在 `userdata/backups/` 创建一致性备份，迁移在单一事务中执行，并在提交前运行 `PRAGMA foreign_key_check`。

回滚 schema v4 时必须停止应用，同时恢复迁移前数据库备份并切回 PR 3 之前的代码；只回退代码、继续使用 v4 数据库不是受支持的回滚方式。
