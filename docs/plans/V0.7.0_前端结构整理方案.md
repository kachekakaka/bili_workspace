# V0.7.0 前端结构整理方案

> 状态：**已批准，待实施**  
> 方案基线：`main` 提交 `6d926c70e98a8e9fed4a1c4b2991f0e52874d991`  
> 目标版本：`0.7.0`  
> Review：[`V0.7.0_前端结构整理方案_REVIEW.md`](V0.7.0_前端结构整理方案_REVIEW.md)

## 1. 目标

V0.7.0 是一次前端内部结构整理版本。它不增加新的业务范围，重点是把当前依赖加载顺序、DOM 后处理和事件抢占的前端，逐页迁移为职责明确、可销毁、可测试的正式模块。

最终目标：

1. 每个页面只有一个正式 renderer；
2. API、认证、Modal、Toast、格式化函数各只有一份实现；
3. 页面切换时旧请求、旧订阅和旧事件处理器能够确定性释放；
4. 任务实时状态只使用一个 `EventSource`；
5. V0.6.2 的账号页、设置页、可搜索选择器和 Modal 直接由正式页面输出，不再事后修改 DOM；
6. 删除 `window.BiliEnhancements`、MutationObserver 页面补丁、捕获阶段兼容事件和版本专用 overlay；
7. 保持 Windows、Docker、权限、搜索请求数量、下载和数据库行为不变。

## 2. 明确非目标

V0.7.0 不做以下工作：

- 不修改下载算法、BBDown 调用语义或画质判定；
- 不修改 SQLite schema，数据库版本继续为 v4；
- 不修改现有 API 路径、请求字段、响应 envelope 或权限矩阵；
- 不增加开放注册、多管理员或每用户独立 Bilibili 凭据；
- 不重构标签业务模型；
- 不引入 React、Vue、Svelte 或其他前端框架；
- 不引入 npm 依赖、打包器、转译器或 Node 运行时生产依赖；
- 不借重构改变页面文案、视觉主题或用户操作顺序；
- 不把桌面端和移动端拆成两套前端。

## 3. 当前事实基线

### 3.1 当前入口

`web/index.html` 当前加载：

- 5 个 CSS 文件；
- `qrcode.min.js`；
- `app.js`；
- 11 个 `enhancements-*.js` / V0.6.2 覆盖脚本；
- `browser-version.js`。

当前入口依赖脚本顺序形成隐式初始化关系。

### 3.2 当前重复实现

`app.js` 和 `enhancements-core.js` 同时实现：

- API client；
- CSRF 处理；
- `toast()`；
- `showModal()`；
- 日期、文件大小、播放量格式化；
- group/quality option 生成；
- 页面或增强页面状态。

任务 SSE 到达后，`app.js` 还需要把任务数据复制进 `window.BiliEnhancements.state.tasks`，说明当前存在双状态同步。

### 3.3 当前补丁特征

代码审计确认当前前端存在：

- 14 个 JavaScript 文件（不含后端代码）；
- 5 个 CSS 文件；
- 10 处 `MutationObserver` 使用；
- 5 处 `stopImmediatePropagation()`；
- 多处 `document`/`window` 全局监听；
- 20 处以上 `BiliEnhancements` 全局引用；
- 账号、设置、作品库、Dashboard 等页面在初次渲染后再次被 overlay 修改。

这些机制在 V0.6.x 中承担兼容作用，但不应继续作为 V0.7 的正式扩展方式。

## 4. Review 后冻结的技术决策

### 4.1 使用原生 ES Modules

新代码使用浏览器原生模块，并统一采用 `.mjs` 扩展名：

```html
<script type="module" src="/assets/app/main.mjs?v=<frontend-version>"></script>
```

选择 `.mjs` 而不是普通 `.js` 的原因：

- Node 的 `--check` 和 `--test` 可以明确按 ESM 解析；
- 不需要增加根目录 `package.json` 或 `type=module`；
- 不影响现有经典脚本的兼容迁移；
- 生产环境仍由 FastAPI 作为普通静态文件提供。

### 4.2 V0.7.0 使用静态 import

V0.7.0 首版使用显式静态 import 和路由注册表，不在本次重构中同时引入页面懒加载。

原因：

- 当前前端体积较小；
- 静态 import 更容易审查依赖图；
- 避免动态 import 缓存、加载失败和路由竞态成为新的变量；
- V0.7 稳定后可另行评估按页懒加载。

### 4.3 不增加构建链

开发、CI 和生产均直接使用仓库中的 `.mjs`、CSS 和 HTML：

- JavaScript 语法：`node --check`；
- 纯模块单元测试：`node --test`；
- DOM 和交互测试：Playwright；
- 不生成 `dist/`，不提交 bundle。

### 4.4 先建立核心，再吸收 overlay

禁止先把 V0.6.2 逻辑继续塞回旧 `app.js`，然后再次拆分。冻结顺序为：

1. 先建立唯一 API、Router、Session、TaskStream 和通用组件；
2. 再切换 App Shell；
3. 页面迁移时同时吸收对应 overlay；
4. 每个 overlay 删除都必须与正式替代实现位于同一 PR。

### 4.5 允许一个临时兼容桥，但必须有删除门禁

迁移期间允许：

```text
window.BiliLegacyApp
window.BiliEnhancements
```

但仅作为旧页面适配器。要求：

- 兼容桥集中在 `web/assets/app/legacy/`；
- 新页面模块不得直接依赖旧全局对象；
- 每个旧全局都要在架构测试中登记删除阶段；
- PR 7 合并前必须全部删除；
- 最终唯一允许的第三方全局是 `window.QRCode`。

## 6. 核心接口

### 6.1 页面生命周期

每个页面导出统一接口：

```javascript
export async function mount(root, context) {
  // 渲染、请求数据、绑定事件

  return {
    dispose() {
      // 释放页面订阅和页面资源
    },
  };
}
```

`context` 至少提供：

```text
api
signal
session
sharedContext
taskStream
modal
toast
navigate
```

约束：

- 页面只能写入传入的 `root`；
- 页面事件优先绑定在 `root`，并使用 `AbortSignal` 自动释放；
- 页面不得直接重写导航、顶栏或其他页面 DOM；
- 页面不得创建永久全局监听；
- `dispose()` 必须幂等。

### 6.2 Router

Router 继续使用现有 hash：

```text
#/dashboard
#/download
#/search
#/library
#/groups
#/tasks
#/users
#/account
#/settings
#/more
```

每次路由切换：

1. 增加 route generation；
2. abort 上一页的 `AbortController`；
3. 调用上一页 `dispose()`；
4. 校验角色与允许页面；
5. mount 新页面；
6. 只有当前 generation 可以提交最终 DOM/状态。

路由跳转和刷新不得依赖 MutationObserver。

### 6.3 API client

唯一 `api()` 实现负责：

- JSON envelope；
- CSRF；
- `credentials: same-origin`；
- `cache: no-store`；
- `AbortSignal`；
- 统一错误消息；
- 401 转换为 `AuthExpiredError`。

API client 不得直接调用 `bootAuth()`，由 App Shell 捕获认证失效并切换登录状态，避免 API 层与 UI 层递归耦合。

### 6.4 Session 和共享上下文

全局只共享：

- 当前网站用户；
- CSRF；
- 服务状态；
- 管理员可见的分组和标签定义；
- 当前前端/服务版本；
- TaskStream。

搜索关键词、作品库筛选、分页、选中项和页面表单草稿归页面自己管理。

共享上下文必须提供：

- 去重的并发加载；
- 明确的失效方法；
- 读取快照；
- 可取消订阅；
- 登出后完整清空。

### 6.5 TaskStream

整个已登录应用最多创建一个：

```javascript
new EventSource('/api/events')
```

TaskStream 负责：

- 连接与重连状态；
- 解析 task payload；
- 保存最新任务和 summary；
- Dashboard/Tasks 多订阅者分发；
- 登出后关闭；
- 会话失效后关闭；
- 不因进入/退出任务页重复创建连接。

## 7. 通用组件要求

### 7.1 Modal

正式 Modal 必须支持：

- 同时只存在一个活动 Modal；
- Escape 关闭（危险操作确认除外）；
- 关闭后焦点返回触发元素；
- 初始焦点；
- backdrop 点击策略；
- 提交期间禁用重复提交；
- `dispose()` 时强制关闭；
- 不使用 `window.prompt()` 或 `window.confirm()`。

### 7.2 SearchableSelect

仅用于用户、分组、任务拥有者等动态数据：

- 少量固定选项继续使用原生 `<select>`；
- 默认阈值保持 8 项；
- 支持键盘、搜索和当前选择标识；
- 原始 `<select>` 或隐藏 input 继续作为表单真值；
- 不通过 MutationObserver 扫描全页自动增强；
- 页面在创建字段时显式调用组件。

### 7.3 Toast

唯一 Toast 服务负责：

- success/warn/error；
- 自动消失；
- 重复消息合并或限流；
- 登出或 App Shell 重置时清理。

## 8. 页面迁移顺序

迁移顺序按风险从低到高冻结：

```text
App Shell / Auth
→ Dashboard / Download / Groups / Users / Account / Settings / More
→ Tasks
→ Library
→ Search
→ 删除兼容层并发布
```

搜索最后迁移，因为它包含请求 generation、AbortController、缓存和下一页预加载；作品库次后，因为它包含播放、批量操作、标签和分组；任务页先于两者，以便尽早建立唯一 SSE。

## 10. 测试方案

### 10.1 静态架构测试

- 禁止新增 `enhancements-*`；
- 禁止新增 `ui-v*` 版本覆盖文件；

### 10.2 Node 单元测试

- hash route 解析和权限回退；
- generation 判定；
- AbortController 生命周期；

6. Settings 高级区展开/折叠；
7. 普通用户无权限 hash 自动回到 Download；
8. 登出后 SSE 关闭、共享状态清空、旧请求中止；
9. 浏览器前端版本和服务版本不一致时给出明确恢复提示。

### 10.4 请求计数门禁

必须保留当前性能边界：

- App 启动不重复请求 auth/status；
- 普通用户不请求 groups/tags/library/search；
- 搜索切换精确/模糊不请求网络；
- Library 一个筛选操作只发起一次列表请求；
- TaskStream 不因页面切换重复连接。

## 11. 代码审查门禁

每个实现 PR 必须说明：

- 替代了哪个旧 renderer/overlay；
- 删除了哪些旧文件；
- 是否新增全局状态、全局监听或观察器；
- 页面 `dispose()` 清理了哪些资源；
- 失败和回滚路径；
- 执行过的验证命令和结果；
- CI run 和结论。

禁止只添加新模块而长期保留相同职责的旧实现。若替代文件不能在同一 PR 删除，必须在 PR 描述中标出明确的删除阶段和兼容原因。

## 12. 回滚策略

V0.7.0 不修改数据库和 API，因此每个 PR 均应支持纯代码回滚：

1. 停止应用；
2. 切回上一个已验证 main/tag；
3. 保留原 `config/`、`userdata/` 和 `downloads/`；
4. 重新运行 `start.bat` 或 Docker 启动；
5. 运行 `verify.bat` 或对应源码验证。

任何阶段都不得用前端重构改变持久化格式。

## 13. 完成定义

V0.7.0 只有同时满足以下条件才算完成：

- PR 1–8 全部合并；
- 旧 `app.js` 和全部 enhancement/overlay 文件删除；
- 单一 App Shell、API client、Session store 和 TaskStream；
- 页面生命周期和路由取消机制生效；
- 无 DOM 后处理型 MutationObserver；
- 无捕获阶段兼容事件；
- 无浏览器原生 prompt/confirm；
- 全量自动化测试和发布门禁通过；
- `main`、tag、Release、README、CHANGELOG 和验收文档一致。

## 14. 实施纪律

- GitHub 仓库是唯一事实来源；
- 每个窗口开始先读取最新 `main`、open PR、feature 分支和 CI；
- 不相信旧聊天中的完成描述；
- 一次只实施一个 PR 阶段；
- 每阶段必须 commit、push、PR、CI green、merge main；
- CI 失败必须读取具体 job/log/artifact 后修复，不得跳过；
- 不允许在回复后假装后台继续执行；
- 没有 commit SHA 不得声称已修改；
- 没有 PR 和 CI 证据不得声称阶段完成。
