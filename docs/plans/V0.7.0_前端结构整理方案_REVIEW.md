# V0.7.0 前端结构整理方案 Review

> Review 对象：[`V0.7.0_前端结构整理方案.md`](V0.7.0_前端结构整理方案.md)  
> Review 基线：`main` 提交 `6d926c70e98a8e9fed4a1c4b2991f0e52874d991`  
> Review 结论：**通过，按本文修正后的阶段顺序实施**

## 1. Review 范围

本次 Review 检查：

- 当前 `web/index.html` 资源加载；
- `app.js` 与 `enhancements-core.js` 公共能力重复；
- Search、Library、Tasks 页面所有权；
- V0.6.2 UI、Settings 和 Library overlay；
- MutationObserver、全局监听和捕获阶段事件；
- 当前 Playwright、前端静态测试和发布 CI；
- 原生 ES Modules 与现有 FastAPI 静态资源/CSP 的兼容性；
- 分阶段回滚和每 PR 可验证性。

## 2. 审计结果

当前事实：

- `index.html` 加载 5 个 CSS 和 14 个 JavaScript 文件；
- `app.js` 约 708 行，负责认证、路由、公共状态和多个页面；
- `enhancements-core.js` 约 371 行，再实现一套 API、状态、Modal 和 Toast；
- 前端共有 10 处 MutationObserver；
- 共有 5 处 `stopImmediatePropagation()`；
- Account、Settings、Dashboard 和 Library 存在初次渲染后的 DOM 修补；
- Tasks 数据需要在两套状态之间同步；
- 现有测试保护脚本加载顺序，说明顺序已经构成运行契约。

结论：启动 V0.7.0 重构是必要且时机合适的，但必须渐进实施。

## 3. 对初始方案的修正

### 3.1 通过：原生模块、零构建依赖

现有应用由 FastAPI 提供同源静态资源，CSP 允许同源脚本，使用原生 ES Modules 可行。

Review 要求模块使用 `.mjs`，避免 Node 对普通 `.js` 的 CommonJS/ESM 判断问题，也避免增加 `package.json`。

### 3.2 修正：V0.7 首版不做动态 import

初始设想按页动态加载，但本次 Review 决定 V0.7 首版使用静态 import。

理由：

- 当前总前端规模不大；
- 重构目标是消除隐式依赖，不应同时增加懒加载故障面；
- 静态依赖更容易被 `node --check`、Review 和架构测试覆盖。

动态 import 作为 V0.7 之后的独立优化，不进入本期。

### 3.3 修正：先 Core/App Shell，后吸收 overlay

初始阶段把“吸收 V0.6.2 overlay”放得过早，容易把新逻辑再次写入旧 `app.js`。

修正为：

```text
架构门禁
→ Core/组件/兼容桥
→ 新 App Shell + 低风险页面 + V0.6.2 正式化
→ Tasks
→ Library
→ Search
→ 删除 Legacy
→ Release
```

### 3.4 增加：兼容桥的边界和截止时间

渐进迁移需要临时兼容，但必须防止兼容层永久存在。

冻结要求：

- 兼容桥只能位于 `web/assets/app/legacy/`；
- 新页面不得调用旧全局；
- PR 7 前删除 `BiliLegacyApp` 和 `BiliEnhancements`；
- 最终架构测试直接拒绝这些 token。

### 3.5 增加：唯一 SSE 是前置核心能力

当前任务状态可能由多个层维护。Review 将 `TaskStream` 提升为 Core 能力，在 App Shell 切换阶段就建立单例，并在任务页迁移后删除旧连接。

必须用 Playwright 或可观测测试验证反复进入 Tasks 不增加 `/api/events` 连接。

### 3.6 修正：CSS 不做无意义大改名

V0.7 的目标是所有权和生命周期，不是视觉重做。

Review 决定：

- 可保留稳定 class 名；
- 删除版本专用 overlay；
- 收口为 tokens/base/components/pages 四层；
- 不要求为了模块化改写全部 HTML class。

### 3.7 增加：AuthExpiredError 解耦

API client 不应在 401 时直接调用认证 UI。Review 要求 API 层抛出明确错误，由 App Shell 统一处理登录失效、关闭 SSE、清空状态和跳转认证页面。

## 4. 风险矩阵

| 风险 | 等级 | 控制措施 |
|---|---:|---|
| 新旧 Router 同时响应 hash | 高 | HTML data attribute 只能启用一个 Shell；架构测试禁止双 boot |
| 新旧 SSE 重复连接 | 高 | Core TaskStream 单例；旧 Tasks adapter 显式禁用自身 EventSource |
| Search 旧响应覆盖新结果 | 高 | route generation + page generation + AbortController 三重门禁 |
| Overlay 删除导致隐藏功能丢失 | 高 | 每个 overlay 删除必须与正式替代和 Playwright 场景同 PR |
| 普通用户误请求管理员 API | 高 | 保留现有 forbidden request Playwright 探针 |
| Modal 重构产生重复提交 | 中 | 单 Modal、提交锁、focus return、重复进入页面测试 |
| ES Module 缓存导致旧文件混用 | 中 | 静态资源 no-cache、前端 build id、版本不一致恢复提示 |
| CSS 收口引发布局回退 | 中 | 五档固定视口 + 无横向溢出/控件重叠 |
| 大 PR 难回滚 | 高 | 固定 8 个阶段，禁止一次性删除全部 legacy |
| 新模块与旧模块长期并存 | 高 | 兼容 token 静态门禁和 PR 7 截止条件 |

## 5. 方案完整性检查

### 功能边界

通过。方案明确不修改下载、数据库、API、权限和标签业务模型。

### 实施顺序

通过。按 Tasks → Library → Search 的风险顺序迁移，Search 最后处理。

### 测试可执行性

通过。项目已有 Python、Node 语法检查和 Playwright；增加 `node --test` 不需要第三方依赖。

### 回滚能力

通过。无数据库迁移，每个阶段可以代码级 revert；持久化目录不变。

### 发布门禁

通过。保留 Python 3.11–3.13、Windows、Docker、Playwright 和完整 pytest。

## 6. 批准条件

方案批准，但实施必须遵守：

1. 不得把所有页面一次性重写到一个 PR；
2. 不得添加 `enhancements-v070-*` 或 `ui-v070-*` 临时覆盖文件；
3. 不得引入 npm 包或打包器；
4. 新页面必须实现 mount/dispose；
5. 每删除一个旧文件，必须在同一 PR 有正式替代和行为测试；
6. CI 失败必须修复，不得降低测试标准；
7. PR 7 未删除全部兼容全局前，不得发布 V0.7.0；
8. Release 只能从全绿的最新 `main` 创建。

## 7. 最终结论

**方案通过。**

推荐立即从 PR 1 开始，先建立架构门禁和 `.mjs` 测试骨架，不直接改生产入口。每个阶段完成后合并到 `main`，下一阶段必须从新的最新 `main` 建分支。
