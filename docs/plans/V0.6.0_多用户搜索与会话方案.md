## 1. 版本目标

V0.6.0 完成四类改造：

1. Bilibili 原始搜索与当前页标题二级筛选分离，并降低搜索请求数量；
2. 增加管理员和普通用户两种角色；
3. 每个用户使用独立网站账号、独立会话 Token、独立任务和导出权限；
4. 统一修正搜索、下载、任务、用户管理及其他页面的响应式布局。

### 1.1 本期纳入

- 单管理员 + 多个普通用户；
- 所有运行模式都要求网站登录；
- Windows 本机初始管理员；
- Docker/NAS 初始化令牌；
- 每用户最多 10 个有效会话 Token；
- 强制首次修改临时密码；
- 普通用户只访问“下载”和“任务”；
- 普通用户只能导出到当前设备；
- 任务所有权、日志权限、SSE 隔离和导出隔离；
- 普通用户终态任务保存 7 天、最多 100 条；
- 普通用户最多 10 个活动任务；
- 管理员按用户筛选、排序和管理任务；
- 原始搜索、精准/模糊标题二级筛选和最多下一页预加载；
- WBI 密钥缓存和搜索请求合并；
- 全站桌面、平板和手机布局验收；
- SQLite 自动备份、迁移和回滚测试。

### 1.2 本期不纳入

- 开放注册；
- 多管理员；
- 匿名公开分享；
- 普通用户访问搜索、作品库、分组、标签、播放或设置；
- 普通用户向媒体库保存文件；
- 每用户独立 Bilibili 账号；
- 可复制的个人 Bearer API Token；
- 自动扫描全部 Bilibili 搜索页；
- 自动生成多码率 HLS；
- 与本方案无关的仓库结构整理。

## 2. 身份与凭据边界

系统中必须明确区分两类凭据：

### 2.1 bili_workspace 网站会话 Token

用于确认当前访问者是谁、属于哪个角色、能查看和操作哪些任务。

- 每个用户使用自己的网站账号和密码；
- 每次登录创建一个独立会话 Token；
- Token 只写入 HttpOnly Cookie；
- 数据库只保存 Token 哈希；
- 任务拥有者只能来自已认证会话，客户端不能指定；
- 本方案中的“每用户最多 10 个有效 Token”指网站会话 Token。

### 2.2 Bilibili 登录凭据

用于服务器访问 Bilibili 搜索、会员画质和下载接口。

V0.6.0 继续使用管理员维护的服务器级 `BBDown.data`：

- 普通用户不能查看或管理 Bilibili 凭据；
- 普通用户网站 Token 不等于 Bilibili Cookie；
- Bilibili Cookie 不返回浏览器、不进入任务 JSON、不进入日志；
- 以后如增加每用户 Bilibili 凭据，另立版本和迁移方案。

## 4. 账号、显示名和密码规则

### 4.1 登录账号

- 长度 3–32；
- 第一个字符必须是英文字母；
- 后续只允许英文字母、数字、点、下划线和短横线；
- 大小写不敏感；
- 创建后普通用户不能修改登录账号。

正则语义：

```text
^[A-Za-z][A-Za-z0-9._-]{2,31}$
```

允许：

```text
zhangsan
user_01
guest-a
```

不允许：

```text
张三
01user
user name
```

### 4.2 中文显示名

- 长度 2–12 个汉字；
- 允许简体和繁体汉字；
- 不允许英文、数字、空格和标点；
- 显示名允许重复；
- 管理员界面始终显示 `显示名（username）`，避免重名混淆。

### 4.3 正式密码

- 长度 10–64；
- 只允许可见 ASCII；
- 至少包含一个英文字母；
- 至少包含一个数字；
- 不允许中文、空格和控制字符；
- 继续使用现有 scrypt 哈希，不保存可逆密码。

`123456` 只允许作为 Windows 回环本机首次管理员登录的受限临时密码。

## 5. 默认管理员与首次登录

### 5.1 Windows 本机全新安装

空数据库第一次启动时自动创建：

```text
登录账号：admin
临时密码：123456
中文显示名：管理员
角色：admin
must_change_password=1
```

限制：

- 临时密码只允许从 `127.0.0.1` 或 `::1` 登录；
- 临时密码登录后只允许访问登录状态、修改密码和退出接口；
- 修改密码前不能下载、搜索、查看任务、进入作品库或设置；
- 初始密码未修改前，应用拒绝切换到非回环监听；
- 密码修改成功后轮换当前会话并撤销其他会话。

### 5.2 Docker、NAS 和公网模式

继续使用现有一次性初始化令牌流程：

- 不开放 `admin / 123456` 远程初始化；
- 初始化令牌只可使用一次；
- 初始化成功后删除令牌；
- 首次设置的正式密码遵守 10–64 位密码规则。

### 5.3 已有安装升级

- 现有唯一用户迁移为 `role=admin`；
- 保留现有登录账号和密码；
- 不重置成 `123456`；
- 缺少显示名时填“管理员”；
- 迁移后旧会话全部失效一次；
- 现有任务、导出和观看记录归属该管理员。

## 6. 会话 Token 设计

### 6.1 每次登录一个独立 Token

同一用户在电脑、手机或不同浏览器登录时分别创建独立会话：

```text
用户 A / 电脑浏览器 -> token A1
用户 A / 手机浏览器 -> token A2
用户 B / 电脑浏览器 -> token B1
```

同一浏览器的多个标签页共享同一个 Cookie，只占一个会话名额。

### 6.2 每用户最多 10 个有效 Token

冻结常量：

```python
MAX_ACTIVE_SESSIONS_PER_USER = 10
```

有效会话定义：

```text
revoked_at IS NULL
并且 expires_at > 当前时间
并且用户未禁用
```

创建第 11 个有效 Token 时，服务端必须在同一个 SQLite 事务中：

1. 创建新会话；
2. 查询该用户全部有效会话；
3. 按 `last_seen_at ASC, created_at ASC` 排序；
4. 保留最近连接的 10 个；
5. 将其余会话设置 `revoked_at` 和撤销原因；
6. 新创建的会话不得被本次清理误删。

这里“连接时间”正式定义为 `last_seen_at`：最久没有成功连接或请求的会话优先失效；时间相同时，最早创建的会话先失效。

### 6.3 最近连接时间

- 每个成功认证的请求都可更新 `last_seen_at`；
- 为减少 SQLite 写入，同一会话最多每 60 秒落库一次；
- SSE 每 30 秒发送一次心跳并重新验证会话；
- 会话因第 11 次登录、禁用或改密失效后，SSE 在下一次心跳关闭。

### 6.4 Token 生命周期

- 绝对有效期默认 30 天；
- `last_seen_at` 不无限延长绝对有效期；
- 过期或已撤销会话不占用 10 个名额；
- Token 使用 `secrets.token_urlsafe(48)` 或同等熵生成；
- 数据库只保存 SHA-256 哈希；
- 明文 Token 不得进入数据库、日志、URL、任务 JSON、前端状态、`localStorage` 或 `sessionStorage`。

### 6.5 Cookie

本地 HTTP：

```text
HttpOnly
SameSite=Lax
Path=/
```

HTTPS：

```text
__Host-bili_session
HttpOnly
Secure
SameSite=Lax
Path=/
```

### 6.6 CSRF

Cookie 会话的以下请求必须验证当前会话独立的 CSRF Token：

```text
POST
PUT
PATCH
DELETE
```

读取接口和 SSE 不要求 CSRF。不能接受其他会话的 CSRF Token。

## 7. 登录、状态和前端认证协议

### 7.1 登录

```http
POST /api/auth/login
```

请求：

```json
{
  "username": "zhangsan",
  "password": "Temporary123"
}
```

登录响应不返回明文会话 Token，只通过 HttpOnly Cookie 设置。响应可返回：

```json
{
  "ok": true,
  "data": {
    "user": {
      "id": "usr_xxx",
      "username": "zhangsan",
      "display_name": "张三",
      "role": "user"
    },
    "permissions": [
      "download:create-device",
      "tasks:read-own",
      "tasks:write-own"
    ],
    "must_change_password": true,
    "csrf_token": "...",
    "session_expires_at": 1780000000,
    "active_session_count": 3
  }
}
```

### 7.2 页面启动

前端先调用：

```http
GET /api/auth/status
```

取得当前用户、角色、权限、强制改密状态、CSRF、会话过期时间和有效会话数量。

前端状态只能保存非敏感资料，不得存在 `state.auth.token`。

### 7.3 强制改密会话

`must_change_password=true` 时只允许：

```text
GET  /api/auth/status
POST /api/auth/password
POST /api/auth/logout
GET  /healthz
```

其他接口返回：

```http
403 Forbidden
```

```json
{
  "ok": false,
  "code": "password_change_required",
  "error": "首次登录必须修改临时密码"
}
```

### 7.4 统一认证上下文

中间件认证完成后写入：

```python
request.state.auth_context = {
    "user_id": "...",
    "username": "...",
    "display_name": "...",
    "role": "admin | user",
    "permissions": [...],
    "session_id": "...",
    "must_change_password": False,
}
```

客户端提交的 `user_id`、`owner_user_id`、`role` 或权限列表一律忽略。

## 8. 会话管理

用户可查看最多 10 个有效会话：

```text
当前设备标记
浏览器 / 系统摘要
登录 IP
登录时间
最近连接时间
过期时间
```

用户操作：

- 退出当前设备；
- 撤销某个其他设备；
- 退出除当前设备外的所有设备。

管理员可：

- 查看某用户有效会话数量；
- 撤销某用户全部会话；
- 重置用户临时密码并撤销全部会话；
- 禁用用户并撤销全部会话。

密码修改后：

- 当前会话 Token 和 CSRF Token 轮换；
- 其他会话全部撤销；
- 原 Token 再使用时返回 401。

退出登录只撤销当前会话，不影响同一用户其他设备。

## 9. 数据库结构与迁移

当前 schema 升级时必须先备份数据库，再在事务中迁移。

### 9.1 users 新增字段

```sql
role                  TEXT NOT NULL DEFAULT 'user'
display_name          TEXT NOT NULL DEFAULT ''
must_change_password  INTEGER NOT NULL DEFAULT 0
created_by            TEXT
last_login_at         REAL
```

### 9.2 sessions 新增字段

```sql
revoked_at     REAL
revoke_reason  TEXT NOT NULL DEFAULT ''
```

新增索引：

```text
(user_id, revoked_at, expires_at, last_seen_at)
```

### 9.3 正式任务记录

新增或迁移为正式任务记录表：

```sql
task_records(
    id,
    owner_user_id,
    destination,
    source_key,
    bvid,
    title,
    status,
    created_at,
    started_at,
    finished_at,
    updated_at,
    payload_json
)
```

索引至少包含：

```text
owner_user_id + created_at
owner_user_id + finished_at
owner_user_id + status
destination + status
```

### 9.4 exports

增加：

```sql
owner_user_id
```

同一作品的有效设备导出判定必须从全局 `source_key` 改为：

```text
owner_user_id + source_key
```

不同用户可以同时导出同一个 BV，互不阻塞。

### 9.5 审计日志

增加会话和目标用户关联字段，但不得记录明文 Token、密码、Bilibili Cookie 或普通用户完整下载 URL。

### 9.6 自动备份与回滚

迁移前使用 SQLite `backup()` 创建：

```text
userdata/backups/bili_workspace-v<旧版本>-YYYYMMDD-HHMMSS.db
```

规则：

- 只保留最近 3 份迁移备份；
- 迁移在事务中完成；
- 任一步失败立即回滚；
- 迁移失败时不开放 HTTP 服务；
- 迁移完成后执行 `PRAGMA foreign_key_check`；
- 现有用户、任务和导出归属原管理员；
- 回滚时停止应用、恢复备份、切回旧版本标签后再启动。

## 10. 任务所有权与权限隔离

每个任务必须包含：

```text
owner_user_id
```

任务创建时只能从当前认证上下文取得用户 ID，前端和 API 请求体不能设置拥有者。

必须执行所有权检查的接口：

```text
任务列表
任务详情
任务日志
暂停
继续
取消
重试
删除
导出状态
导出文件下载
SSE
```

普通用户：

- 只能看到自己的任务；
- 只能操作自己的任务；
- 只能下载自己的导出文件；
- SSE 只推送自己的任务。

管理员：

- 可以查看和操作所有用户任务；
- 可以按用户筛选、排序和分组展示；
- 管理员重试普通用户任务时，任务仍归属原普通用户。

## 13. 任务保留与清理

冻结常量：

```python
NORMAL_USER_TASK_RETENTION_DAYS = 7
NORMAL_USER_TASK_HISTORY_LIMIT = 100
NORMAL_USER_ACTIVE_TASK_LIMIT = 10
ADMIN_TASK_HISTORY_LIMIT = 500
```

### 13.1 普通用户终态任务

满足任一条件就删除：

```text
finished_at 距今超过 7 天
该用户终态任务超过 100 条
```

超过 100 条时按 `finished_at` 最旧优先清理。

同步清理：

```text
任务记录
任务日志
失效导出元数据
残留临时文件引用
包含作品信息的普通用户下载审计记录
```

不因保留规则删除：

```text
queued
running
正在取消
尚未完成传输的导出文件
```

### 13.2 管理员任务

管理员自己的终态任务最多保留 500 条，不受普通用户 7 天规则影响。

### 13.3 活动任务限制

每个普通用户最多 10 个 `queued + running` 任务。达到上限后拒绝新增第 11 个活动任务。

登录 Token 上限 10 和活动任务上限 10 是两个独立常量，不得混用。

### 13.4 导出文件

任务记录保留 7 天不代表临时文件保留 7 天：

- 浏览器完整接收后立即删除文件；
- 传输中断时按现有导出 TTL 保留；
- TTL 到期后自动清理文件；
- 文字任务记录可继续保留到 7 天。

## 15. 搜索行为与性能

### 15.1 基本原则

Bilibili 原始搜索和标题二级筛选必须分开。

搜索区只负责：

```text
B站关键词
B站排序：综合 / 播放最多 / 最新发布
搜索
刷新B站结果
```

结果区提供当前页标题二级筛选：

```text
不筛选
精准：标题包含全部词
模糊：标题包含任意词
标题筛选词
```

行为：

- “搜索”只请求当前一页 Bilibili 原始结果；
- 原始结果保存在浏览器页缓存；
- 精准和模糊只过滤当前页标题；
- 切换筛选模式不请求网络；
- 修改标题筛选词不请求网络；
- UP 主和 BV 号不参与二级匹配；
- “刷新B站结果”才清除当前页服务端缓存；
- 不允许页面暗示已经扫描全部搜索页。

摘要示例：

```text
B站当前页返回 20 条 · 标题筛选后 6 条 · 第 1 / 35 页
```

当前页没有匹配项时显示：

```text
本页没有标题匹配项，可查看下一页；系统不会自动抓取全部页面。
```

### 15.2 页码和预加载

十个页码按钮只是导航，不代表抓取十页数据。

冻结加载规则：

```text
立即加载当前页
当前页渲染完成后，在浏览器空闲时最多预加载下一页 1 页
不预加载第三页
```

约束：

- 修改 B站关键词或排序时取消旧请求和旧预加载；
- 省流量模式不预加载；
- 当前页失败时不预加载；
- 切换精准/模糊时不触发预加载；
- 筛选后为空时不连续抓后续页面。

### 15.3 服务端优化

- WBI 密钥按 Bilibili Cookie 指纹缓存 10 分钟；
- 签名失效时清除密钥并最多重试一次；
- 原始搜索页缓存 3 分钟；
- 缓存键只包含 B站关键词、B站排序、页码和 Cookie 指纹；
- 精准/模糊模式不得进入原始搜索缓存键；
- 普通搜索允许缓存命中；
- 只有“刷新B站结果”清除当前页缓存；
- 搜索响应直接附带标签、下载状态和删除状态；
- 删除浏览器额外的标签批量请求；
- 前端使用 `AbortController` 取消旧搜索；
- 旧响应不能覆盖新关键词结果。

性能目标：

```text
切换精准/模糊：0 次网络请求
首次搜索：最多 WBI + 搜索 2 次 Bilibili 请求
WBI 缓存命中：1 次 Bilibili 搜索请求
搜索页缓存命中：不访问 Bilibili
自动预加载：最多下一页 1 次
```

## 20. 测试方案

### 20.1 会话 Token

1. 一个用户可以同时拥有 10 个有效 Token；
2. 第 11 次登录后有效 Token 总数仍为 10；
3. 最久未连接的 Token 被撤销；
4. 最早创建但刚使用过的 Token应保留；
5. `last_seen_at` 相同时最早创建的 Token 被撤销；
6. 新创建的第 11 个 Token不会被误撤销；
7. 过期 Token 不占名额；
8. 已撤销 Token 不占名额；
9. 并发登录事务结束后仍不超过 10 个有效 Token；
10. 被挤掉的 Token 访问 API 返回 401；
11. 被挤掉的 Token 对应 SSE 在心跳时关闭；
12. 修改密码撤销其他 Token 并轮换当前 Token；
13. 管理员重置密码撤销该用户全部 Token；
14. 禁用用户撤销全部 Token；
15. 退出登录只撤销当前 Token；
16. “退出其他设备”保留当前 Token；
17. 数据库、API 响应和日志中不存在明文 Token。

### 20.2 Cookie 与 CSRF

1. Cookie 包含 HttpOnly；
2. HTTPS Cookie 包含 Secure；
3. HTTPS 使用 `__Host-bili_session`；
4. Cookie 写请求缺少 CSRF 返回 403；
5. 使用其他会话的 CSRF 返回 403；
6. 读请求不要求 CSRF；
7. 失效 Token 返回 401；
8. 临时密码会话访问普通 API 返回 `password_change_required`。

### 20.3 账号规则

1. 用户名只允许规定 ASCII 格式；
2. 用户名大小写不敏感；
3. 中文账号和数字开头账号被拒绝；
4. 中文显示名规则正确；
5. 英文、数字、空格或标点显示名被拒绝；
6. 正式密码必须包含英文和数字；
7. 正式密码不能包含中文、空格和控制字符；
8. 已有管理员升级不重置密码；
9. Windows 回环新安装创建 `admin / 123456`；
10. 局域网不能使用未修改的默认密码；
11. Docker 仍要求初始化令牌。

### 20.4 RBAC 与越权

1. 普通用户不能访问概览、搜索、作品库、设置和 Bilibili 账号；
2. 普通用户不能创建或修改分组；
3. 普通用户手工提交 `destination=library` 仍被强制改成设备导出；
4. 普通用户提交他人 `owner_user_id` 被忽略；
5. 用户 A 看不到用户 B 的任务和日志；
6. 用户 A 不能操作或下载用户 B 的任务和导出文件；
7. 管理员可以查看和操作全部用户任务；
8. 两个用户可以同时导出同一个 BV；
9. 普通用户 SSE 只收到自己的任务；
10. 重试普通用户任务时拥有者保持不变。

### 20.5 任务保留

1. 普通用户第 101 条终态任务写入后只保留最新 100 条；
2. 7 天前终态任务被删除；
3. 6 天 23 小时任务保留；
4. queued、running 和正在取消的任务不被时间规则删除；
5. 删除任务时同步删除日志和失效导出元数据；
6. 管理员任务不受普通用户 7 天规则影响；
7. 一个用户的清理不能删除其他用户任务；
8. 普通用户活动任务达到 10 个后拒绝第 11 个；
9. 终态任务不占活动任务名额；
10. 传输中的导出文件不被任务历史清理误删。

### 20.6 搜索后端

1. 每次只请求一页、每页 20 条；
2. 首次搜索最多产生 WBI + 搜索两次 Bilibili 请求；
3. WBI 缓存命中后只有一次搜索请求；

5. “刷新B站结果”才清除当前页缓存；
6. 精准/模糊不进入原始搜索缓存键；
