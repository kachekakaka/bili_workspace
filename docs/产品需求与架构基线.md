# bili_workspace 产品需求与架构基线

本文件是后续开发、审查和发布的长期依据。新增需求落地时，应同时更新本文件、测试和对应使用文档，避免功能只存在于聊天记录中。

## 1. 产品定位

`bili_workspace` 同时提供：

- Windows 本机下载与媒体库；
- QNAP NAS / Docker 长期运行服务；
- 手机、平板和桌面浏览器访问；
- Bilibili 搜索、下载、任务管理、标签、分组、在线播放和设备导出。

仅用于用户有权下载、保存和播放的内容。

## 2. 搜索

搜索必须把 Bilibili 原始搜索和浏览器当前页标题二级筛选分开：

- 搜索区只提交 B站关键词、B站排序和页码；
- “不筛选”直接显示 Bilibili 当前页原始结果；
- “精准”仅保留当前页标题包含全部拆分词的作品；
- “模糊”仅保留当前页标题包含任意拆分词的作品；
- 精准/模糊只匹配标题，UP 主名称和 BV 号不得让不匹配的标题通过；
- 切换二级筛选模式或修改标题筛选词不得发起网络请求。

加载和缓存边界：

- 点击搜索只立即加载当前页，每页固定 20 条；
- 当前页成功渲染后，浏览器空闲时最多预加载下一页 1 页，不预加载第三页；
- 省流量模式、当前页失败或查询已变更时不预加载；
- 页码按钮只用于导航，不代表已经抓取对应页面；
- WBI 密钥按 Bilibili Cookie 指纹缓存 10 分钟，签名失效时清除并最多重试一次；
- 原始搜索页缓存 3 分钟，键只包含关键词、B站排序、页码和 Cookie 指纹；
- 只有“刷新B站结果”清除并重取当前页，二级筛选不进入服务端缓存键；
- 前端必须取消旧请求，旧响应不得覆盖新关键词或新排序结果。

搜索页还必须：

- 显示标题、BV、UP 主、封面、时长、播放量和发布信息；
- 在同一搜索响应中取得标签、未下载、排队中、下载中、已下载、失败、已取消和已删除状态；
- 默认提供“屏蔽已下载和已删除”；
- 支持 Bilibili 原作品页直达、跨页选择、批量标签、画质预览和批量下载；
- 创建下载任务后停留在当前页面，不强制跳转任务中心；
- 摘要同时显示 Bilibili 当前页原始数量和标题筛选后数量，且不得暗示已经扫描全部页面。

## 3. 下载与清晰度

- 下载前预览清晰度、分辨率、编码、帧率、分 P 和估算大小；
- 支持最低清晰度与明确目标档位；
- 下载前预检和下载后实际码流复核必须同时通过；
- 不允许任务显示成功却静默保存低于门槛或与指定档位不一致的文件；
- 下载核心应保持可替换，避免与已停止维护的单一第三方工具深度耦合；
- 普通下载和搜索下载成功创建任务后只提示结果，不强制改变当前页面。

## 4. 任务中心

任务中心必须支持：

- SSE 实时更新，网络异常时可重新连接；
- 标题、BV、分组、目标、状态、阶段、时长和分 P 信息；
- 在 BBDown 输出可解析时显示真实百分比、当前大小、总大小、速度和 ETA；
- 单任务和批量暂停、继续、取消、重试、删除；
- 一键清理失败和已取消任务；
- 失败任务修改最低清晰度或指定画质后重新下载；
- 重试复用原任务 ID，不新增重复任务记录；
- 暂停后继续可复用任务记录，但不得伪装成未经验证的字节级断点续传；
- 日志保留原始诊断信息，同时对敏感字段脱敏；
- 任务、日志、SSE 和设备导出按 `owner_user_id` 隔离，普通用户只能访问本人数据；
- 普通用户最多 10 个活动任务，终态任务保留 7 天且最多 100 条；管理员终态任务最多 500 条；
- 管理员任务查询由 SQLite 按用户、状态、目标、关键词和时间字段筛选排序。
- 普通用户侧边栏和手机底栏只显示下载与任务；下载页不显示媒体库目标、分组、标签或 force。
- 顶部用户菜单提供显示名、账号、改名、改密、登录设备和退出；管理员用户管理页桌面用表格、手机用卡片。

## 5. 分组

- 下载时可选择已有分组；
- 可在下载表单内立即创建并选中新分组；
- 提供独立分组页，可查看作品数量、空间、任务和最近更新；
- 支持重命名、合并、移动作品、批量移动和删除空分组；
- 作品卡片和播放详情均可直接修改分组；
- 显示名称与物理目录标识解耦，普通重命名不搬动大型文件；
- 删除分组不得隐式删除媒体文件。

## 6. 标签和删除历史

- 标签定义来自 `config/tags.json`，模板为 `config/tags.json.default`；
- 标签关系保存在 SQLite，不依赖扫描媒体文件；
- 搜索结果和作品库作品均可打标签；
- 作品库支持标签 chip、分组 chip 和“无标签”筛选；
- 默认标签使用可区分配色，用户自定义颜色优先；
- 删除作品时，作品库记录和媒体文件按用户选择删除；
- 删除历史写入独立 `deleted_media` 记录，不使用普通标签代替；
- 删除历史不在作品库显示，但搜索结果可标记为“已删除”；
- 主动重新下载成功后，可清除对应删除历史并恢复为“已下载”。

## 7. 媒体库与播放

- 手机、平板和桌面浏览器均可使用；
- 支持 `HEAD`、字节 `Range`、`206`、`416`、`ETag`、拖动和断点续播；
- 服务端不得把整个大文件一次性读入内存；
- 默认优先直接播放原始文件；
- 浏览器不兼容时由用户手动生成 H.264/AAC MP4 兼容副本；
- 兼容副本不得覆盖原始高质量文件；
- 支持多分 P、播放位置和最近观看；
- 作品库使用数据库服务端筛选、分页和排序；
- 排序支持下载时间、最近观看、标题、时长、文件大小、分组和标签的升降序；
- 作品封面应完整显示，不应无提示裁掉主体。

## 8. 导出到当前设备

下载目标包括：

1. 永久保存到 NAS / 本机媒体库；
2. 临时下载后导出到当前浏览器设备。

设备导出规则：

- NAS 临时下载、混流和必要打包完成后再提供下载；
- 使用流式响应，不把大文件整体载入内存；
- 服务器完整发送全部响应字节后立即删除专属临时文件；
- 浏览器中断、断网或未读完时不立即删除，允许重试；
- 中断文件由可配置 TTL 清理兜底；
- 只能通过数据库对象 ID 访问，浏览器不能提供任意服务器路径；
- 清理逻辑只能作用于设备导出专属临时目录，不得触碰媒体库；
- 有效设备导出按 `owner_user_id + source_key` 判定，不同用户可同时导出同一作品。

## 9. 网站账号、会话与 Bilibili 扫码

- Windows 本机、源码、Docker/NAS 和公网模式都要求网站账号登录；
- Windows 回环全新安装自动创建 `admin / 123456` 临时管理员，仅允许本机首次登录并强制改密；
- Docker/NAS 和非回环全新安装继续使用一次性初始化令牌，不开放远程默认密码；
- 网站账号分为一个管理员和多个普通用户，用户名、中文显示名和正式密码遵守 V0.6 固定规则；
- 每次登录创建独立 HttpOnly 会话 Token，数据库只保存 SHA-256 哈希；
- 每用户最多 10 个有效会话，第 11 次登录在同一 SQLite 事务中按 `last_seen_at`、`created_at` 淘汰最久未连接会话；
- 写请求校验当前会话独立 CSRF Token；改密轮换当前 Token/CSRF 并撤销其他会话；
- 用户可管理自己的登录设备，管理员可创建、禁用、重置普通用户及撤销其会话；
- 需要登录限速、安全 Cookie、可信 Host/代理和不含明文凭据的审计日志；
- 网页可创建并显示 Bilibili 登录二维码；完整 Bilibili Cookie 只保存在服务器秘密目录，不返回浏览器、不进入日志和发布包；
- 普通用户任务、日志、SSE 和导出按所有者隔离；普通用户前端只显示下载和任务，管理员提供用户管理和按用户任务视图。

## 10. 网络与域名

- 监听地址不能写死；支持 `127.0.0.1`、`0.0.0.0`、指定局域网 IP、IPv6 和合法主机名；
- 端口支持 `1–65535`，包括 3389，但应提示与 Windows 远程桌面冲突的可能；
- 设置页应展示可用于手机访问的局域网地址；
- 多网卡环境应列出可用 IP，而不是只假定一个地址；
- 使用系统或手机代理时，应提示把局域网网段和应用地址设为直连；
- 公网推荐使用独立 HTTPS 子域名和 QNAP 反向代理；
- 公网只开放 443，不建议直接裸露应用端口；
- 转发头只信任明确的反向代理来源 IP。

## 11. 配置模板与升级

Git 跟踪：

```text
.env.default
config/config.json.default
config/runtime.env.default
config/tags.json.default
docker/.env.default
```

Git 不跟踪对应实际文件。启动和更新必须：

- 实际配置不存在时从同名 `.default` 创建；
- JSON 配置递归补入模板新增而实际缺少的字段；
- dotenv 配置只追加缺失变量；
- 保留已有值、额外字段、空字符串、`0`、`false` 和用户选择；
- 使用原子写入并保留备份；
- 拒绝符号链接配置目标；
- 损坏配置不得被模板静默覆盖；
- 需要迁移时提供显式版本与可回滚迁移。

## 12. 持久化边界

Windows / 源码运行：

```text
config/      配置和标签定义
userdata/    SQLite、任务快照、删除历史、索引、日志、缓存和临时状态
downloads/   永久媒体文件
```

Docker 容器固定使用：

```text
/data/config     配置和 Bilibili 凭据
/data/userdata   SQLite、任务、删除历史、索引、日志、缓存和临时状态
/downloads       永久媒体文件
```

宿主机目录通过 `docker/.env` 的 `CONFIG_DIR`、`USERDATA_DIR` 和 `MEDIA_DIR` 映射。更新 Git、重建镜像或重建容器不得清空这些目录。

容器应：

- 非 root 运行；
- 不使用 privileged；
- 不挂载 Docker socket；
- 根文件系统只读；
- 丢弃不需要的 capabilities；
- 有健康检查、日志轮转和优雅停止。

## 13. 性能与可靠性

- 任务变化优先通过 SSE 推送，保留重新连接能力；
- 作品库使用数据库服务端筛选、排序和分页；
- 封面懒加载并受控缓存；
- 下载、转码和缩略图并发必须有上限；
- 开始任务前检查临时盘和媒体盘剩余空间；
- 任务、分组、作品库、标签、删除历史和观看记录重启后保留；
- 运行中任务在异常重启后标记中断或恢复排队，不能误判成功；
- 日志、封面缓存、兼容副本和导出临时文件必须有容量或 TTL 边界。

## 14. Git 与发布

- GitHub `main` 必须直接包含可审查源码树，禁止使用 Base64 分片或 Actions 自解包恢复源码；
- 正常 CI 执行源码结构、编译、静态检查、测试和前端语法检查；
- 实际配置、凭据、数据库、媒体、日志和缓存不得提交；
- `.venv` 不提交，因为包含绝对路径、Python ABI 和平台相关入口；
- Windows 通过仓库内校验过的便携运行包恢复 Python、BBDown 和 FFmpeg；
- 运行资产必须固定版本和 SHA-256，内部还需逐文件清单；
- 每次重要发布保留源码 ZIP、完整 Git Bundle、Windows 恢复包、校验文件和验证报告。

## 15. 当前明确不纳入 V0.5.x

- 开放注册；
- 匿名公网分享；
- 多管理员角色；
- 自动为所有媒体生成多码率 HLS；
- 强依赖 NAS GPU；
- 自动删除原始媒体；
- Plex/Jellyfin 深度集成；
- 多节点任务调度。
