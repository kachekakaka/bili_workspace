# bili_workspace V0.5.0 发布说明与验证报告

发布日期：2026-07-17  
版本定位：Windows 本地下载器 + QNAP/NAS 私人媒体库 + 浏览器在线播放与一次性设备导出

## 1. 本版范围

V0.5.0 在 V0.4.0 的分组、标题/BV 双标识、清晰度预览和低清晰度硬拦截基础上，完成以下架构升级：

- Windows 本地模式与 NAS/Docker 服务器模式共用一套后端、前端和数据模型；
- SQLite WAL 持久化管理员、会话、逻辑分组、媒体库、播放进度、任务快照、设备导出和审计日志；
- 响应式作品库、分组管理、任务中心、账号页和设置页；
- 浏览器标准 Range/HEAD/206/416 媒体流，可拖动、继续观看和下载原文件；
- 原文件优先播放，不兼容时由用户手动生成 H.264/AAC/MP4 兼容副本；
- 下载目标可选“保存到媒体库”或“导出到当前设备”；
- 设备导出完整发送后立即清理服务器临时产物，中断则保留到 TTL 到期或用户手动放弃；
- 下载时可搜索选择分组，也可原地新建；分组页可浏览、重命名、合并和删除空分组；
- NAS 模式强制管理员认证、CSRF、可信 Host/代理、安全 Cookie、登录节流、审计、改密和其他会话撤销；
- 网页 Bilibili 扫码登录，完整 Cookie 只写入服务器 `BBDown.data`；
- QNAP Docker 配置包含四个目录映射、非 root 用户、只读根文件系统、能力删除、健康检查和日志轮转。

## 2. 设备导出的准确语义

```text
NAS 临时目录下载和混流
→ 浏览器以流式响应接收
→ 应用完成全部响应字节发送
→ 删除导出文件、源产物和临时索引
```

- 不把大文件整体读入内存；默认按 1 MiB 分块。
- HEAD 请求不会触发清理。
- 浏览器中断或响应迭代未完成时不会触发完成清理。
- 中断文件默认保留 24 小时，可重新下载或手动放弃。
- HTTP 只能证明服务器完成响应发送，不能证明浏览器最终成功写入设备磁盘；重要文件仍建议先保存到媒体库。

## 5. 当前验证环境限制

以下项目不能在本次 Linux 构建环境中冒充为已完成：

- 没有 Docker/Podman/Buildah，因此没有实际构建或运行 Docker 镜像，也不提供预构建镜像 tar；
- 构建环境 DNS 无法解析 GitHub，未现场下载 Linux BBDown 资产；Docker 构建会在你的 NAS 上下载固定版本并执行 `BBDown --help` 冒烟测试；
- 不能原生执行 Windows EXE；Windows 包保留 `verify.bat`，会在你的电脑实际启动 BBDown 和 FFmpeg；
- 浏览器自动化访问本地地址受环境管理策略阻止，因此未声明 Chromium 交互测试通过；接口、Range 流和 JavaScript 语法已自动验证；
- 真实 Bilibili 搜索、二维码扫码、下载、混流、QNAP 目录权限、域名证书和反向代理必须在你的实际网络与设备上验收。

用户此前已确认 V0.3 Windows 实机搜索、下载、混流、播放和打开目录正常；V0.5 新增的管理员、作品库、设备导出和 QNAP 路径仍应按验收文档检查。

## 6. 发布前建议验收

Windows：

1. 解压到新目录，运行 `verify.bat`；
2. 运行 `start.bat`；
3. 新建分组，下载一条 1080P 公开合法样本；
4. 在作品库播放并拖动进度；
5. 创建设备导出，完整下载后确认临时状态变为已清理；
6. 中断另一条大文件导出，确认仍可重试。

QNAP：

1. 修改 `.env` 四个目录、PUID/PGID、HTTPS 域名、可信 Host 和真实代理来源 IP；
2. 运行 `docker/verify-config.sh`；
3. 构建并启动容器，确认健康状态；
4. 初始化管理员并更换密码；
5. 配置 QNAP HTTPS 反向代理，公网只开放 443；
6. 网页扫码登录 Bilibili；
7. 验证 NAS 保存、在线播放、兼容副本、设备导出和重启恢复。
