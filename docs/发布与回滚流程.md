# 发布、更新与回滚流程

## 1. GitHub 发布纪律

源码必须以普通 Git 提交直接存在于 `main`。禁止通过压缩分片、Base64 或运行时恢复源码。

每个阶段：

1. 从最新 `main` 创建功能或 release 分支；
2. commit 并 push；
3. 创建 PR；
4. 等待全部 required validation 通过；
5. 合并 `main`；
6. 仅在 main 同一 SHA 的 CI 和 UI workflow 均成功后创建 tag/Release。

V0.7.0 使用 `.github/workflows/release-v070.yml` 自动执行最后一步。workflow 会核对同一 main SHA 的两套验证、应用版本和前端资源批次，幂等创建 tag `v0.7.0` 与 GitHub Release。

GitHub 会阻止由 `GITHUB_TOKEN` 推送的 tag 再递归触发其他 workflow，因此发布 workflow 不依赖 tag-push 的隐式连锁；它会在 Release 建立后**显式 dispatch** `docker-image.yml`，传入 `release_tag=v0.7.0`。Docker workflow 从不可变 tag 检出源码，并发布 `linux/amd64`、`linux/arm64` 的 `v0.7.0` manifest。已存在排队、运行中或成功的同名发布镜像 run 时不会重复 dispatch。

## 2. 发布前验证

```bash
python tools/verify_source.py
python tools/check_markdown_links.py
python -m compileall -q app tests tools docker
python -m ruff check --no-cache app tests tools docker
python -m pytest -q -p no:cacheprovider
find web -type f \( -name '*.js' -o -name '*.mjs' \) -print0 | sort -z | xargs -0 -n1 node --check
node --test tests/frontend/*.test.mjs
```

还必须通过：

- Playwright 五档固定视口；
- Windows `verify.bat`；
- Docker build 和镜像内版本冒烟；
- 数据库迁移/回滚、持久化和权限测试；
- Markdown 内部链接与敏感信息边界。

## 3. 发布前文档

检查：

- `README.md`、`CHANGELOG.md`；
- `docs/README.md`、`docs/需求落实清单.md`；
- 当前版本功能与验收文档；
- Windows、Docker、备份、迁移和回滚路径；
- 版本号、frontend build、tag 和 Release notes。

持久化边界固定为：

```text
Windows：config/、userdata/、downloads/
Docker：/data/config、/data/userdata、/downloads
```

## 4. Windows 更新

推荐运行：

```bat
update.bat
```

或：

```bat
git pull --ff-only origin main
verify.bat
start.bat
```

`verify.bat` 会准备仓库内便携运行时，并运行 Python、Ruff、pytest、`.js/.mjs` 和 Node tests。实际配置、数据库、媒体和解压目录不会被快进更新覆盖。

## 5. Docker / QNAP 更新

```bash
git pull --ff-only origin main
./docker/build-and-start.sh
```

或直接拉取固定 tag：

```text
ghcr.io/kachekakaka/bili_workspace:v0.7.0
```

保持 `CONFIG_DIR`、`USERDATA_DIR`、`MEDIA_DIR` 不变。默认镜像和本地构建不得改变容器内 `/data/config`、`/data/userdata`、`/downloads`。

## 6. 数据库迁移

所有数据库迁移必须满足一致性备份、单事务、失败回滚、外键检查、未来 schema 拒绝降级和最近 3 份备份。

V0.6.0 已将数据库升级到 schema v4。V0.7.0 继续使用 schema v4，**不执行新的数据库迁移**。由 v0.6.2 升级到 v0.7.0 时可直接复用原 `userdata/`。

## 7. 代码回滚

先停止应用并备份现场：

```bash
git fetch --tags
git checkout <已验证标签或提交>
```

V0.7.0 回滚到 v0.6.2：

1. 停止服务；
2. 备份当前 `config/`、`userdata/` 和媒体目录；
3. 切换 `v0.6.2`；
4. 保持持久化目录不变；
5. 运行对应版本 `verify.bat` 或源码验证；
6. 核对用户、会话、任务、导出、作品、标签、删除历史和播放记录。

V0.7.0 未改变数据库或 API，正常回滚无需数据库降级。

## 8. 数据恢复

安全顺序：

1. 停止应用/容器；
2. 保存当前故障现场；
3. 恢复同一时间点的数据库、WAL/SHM、索引和媒体；
4. 切换匹配的代码版本；
5. 启动并核对全部业务数据。

只恢复数据库或只恢复媒体都可能造成状态不一致。

## 9. 发布资产

GitHub Release 自动提供源码归档。用户还应将下列恢复资产保存到自己的 NAS 或云盘：

```text
完整 Git Bundle
Windows 完整恢复包
QNAP Docker 部署说明
SHA256SUMS.txt
发布验证报告
```

GitHub 不应是唯一备份位置。
