# 发布、更新与回滚流程

## 1. 正常 GitHub 发布

源码必须以普通 Git 提交直接存在于 `main`。禁止通过压缩分片、Base64 文本或 GitHub Actions 运行时恢复源码。

功能开发建议：

1. 从最新 `main` 创建 `agent/*` 分支；
2. 提交只属于本次需求的文件；
3. 创建 PR；
4. 等待 Python 3.11、3.12、3.13、Ruff、pytest 和前端 JavaScript 语法检查全部通过；
5. 合并到 `main`；
6. 用户通过 `git pull --ff-only origin main` 更新。

发布前本地验证：

```bash
python tools/verify_source.py
python -m compileall -q app tests tools docker
python -m ruff check --no-cache app tests tools docker
python -m pytest -q -p no:cacheprovider
find web -type f -name '*.js' -print0 | xargs -0 -n1 node --check
```

然后创建提交和版本标签。需要替换已知损坏远端历史时，只能使用带明确远端 SHA 的 `--force-with-lease`，不能使用无条件 `--force` 覆盖未知提交。

## 2. 发布前文档检查

每次重要功能合并至少检查：

- `docs/README.md` 是否指向当前文档；
- `docs/产品需求与架构基线.md` 是否包含新边界；
- `docs/需求落实清单.md` 是否反映当前能力；
- `docs/V0.6功能与验收.md` 是否有对应验收步骤；
- 部署、备份和目录路径是否仍与代码一致；
- 带版本号的历史报告是否明确标记为历史状态。

当前持久化边界必须写成：

```text
Windows：config/、userdata/、downloads/
Docker：/data/config、/data/userdata、/downloads
```

## 3. Windows 更新

推荐：

```bat
update.bat
```

或手工执行：

```bat
git pull --ff-only origin main
start.bat
```

`start.bat` 会校验并解压仓库内 `vendor/windows/*.pack`，同步配置并启动。实际配置、数据库、媒体和第三方工具解压目录均由 `.gitignore` 排除，不会被快进更新覆盖。

更新后需要完整自检时运行：

```bat
verify.bat
```

## 4. Docker / QNAP 更新

```bash
git pull --ff-only origin main
./docker/build-and-start.sh
```

三个宿主机映射目录必须保持不变：

```text
CONFIG_DIR
USERDATA_DIR
MEDIA_DIR
```

重大数据库迁移前至少备份 `CONFIG_DIR`、`USERDATA_DIR` 和媒体索引/文件。默认 GHCR 镜像更新时也不得改变这三个映射路径。

## 5. 数据库迁移

数据库迁移必须满足：

- 启动时幂等；
- 目标文件或字段已存在时不重复覆盖；
- 主数据库、WAL 和 SHM 同时考虑；
- 旧索引和旧任务日志迁移后仍可回滚；
- 标签关系、删除历史和任务快照不得静默丢失；
- 新增表应由应用启动自动创建；
- schema 升级前使用 SQLite `backup()` 在 `userdata/backups/` 创建一致性备份；
- 迁移在单一事务中完成，失败回滚且不开放 HTTP 服务；
- 迁移成功后执行 `PRAGMA foreign_key_check`，只保留最近 3 份迁移备份；
- 数据库 schema 高于当前程序支持版本时必须拒绝启动，禁止自动降级。

当前删除历史保存在独立 `deleted_media` 表中，不应通过普通标签代替。

## 5.1 V0.6.0 schema v4 升级与回滚

V0.6.0 使用 schema v4。升级会先在 `userdata/backups/` 创建 `bili_workspace-v<旧版本>-YYYYMMDD-HHMMSS.db`，再在事务中迁移，并执行 `PRAGMA foreign_key_check`。旧会话在 v2→v3 时失效一次；v3→v4 不重复撤销有效会话。

需要回滚到 V0.6.0 之前的代码时：

1. 停止应用或容器；
2. 复制当前整个 `userdata/` 作为故障现场；
3. 从 `userdata/backups/` 恢复与旧代码匹配的数据库；
4. 切换到已验证标签或提交；
5. 保持原 `config/` 和 `downloads/` 映射，再启动并核对用户、任务、导出和媒体。

禁止让旧程序直接打开 schema v4 数据库。

## 6. 代码回滚

先停止应用并备份当前数据，再执行：

```bash
git fetch --tags
git checkout <已验证标签或提交>
```

Docker 重新构建后仍使用同一组持久化目录。若版本包含数据库迁移，必须结合对应迁移说明和数据库备份回滚，不得只回退代码而忽略数据结构。

Windows 回滚后运行：

```bat
start.bat
verify.bat
```

Docker 回滚后运行：

```bash
./docker/build-and-start.sh
```

## 7. 数据回滚

安全顺序：

1. 停止应用或容器；
2. 备份当前 `config`、`userdata` 和媒体目录；
3. 恢复同一时间点的数据库、WAL/SHM、索引和媒体；
4. 切换到匹配的代码版本；
5. 启动并核对作品、标签、删除历史、任务、分组和播放记录。

只恢复数据库而不恢复媒体/索引，或只恢复媒体而不恢复数据库，都可能造成状态不一致。

## 8. 发布资产

源码历史不保存不可移植的 `.venv`。Windows 便携运行资产以仓库内经过校验的 pack 和清单形式交付；正式发布仍应额外保存：

```text
源码 ZIP
完整 Git Bundle
Windows 完整恢复包
QNAP Docker 部署说明
SHA256SUMS.txt
发布验证报告
```

Git Bundle、Windows 完整恢复包和 SHA-256 清单应额外保存到用户自己的 NAS 或云盘，GitHub 不应是唯一备份。
