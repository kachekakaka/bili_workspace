# 发布、更新与回滚流程

## 正常 GitHub 发布

源码必须以普通 Git 提交直接存在于 `main`。禁止通过压缩分片、Base64 文本或 GitHub Actions 运行时恢复源码。

发布前：

```bash
python tools/verify_source.py
python -m compileall -q app tests tools docker
python -m ruff check --no-cache app tests tools docker
python -m pytest -q -p no:cacheprovider
find web -type f -name '*.js' -print0 | xargs -0 -n1 node --check
```

然后创建提交和版本标签。需要替换已知损坏远端历史时，只能使用带明确远端 SHA 的 `--force-with-lease`，不能使用无条件 `--force` 覆盖未知提交。

## Windows 更新

```text
update.bat
```

它只允许干净的受 Git 管理工作区，使用 `git pull --ff-only`，随后同步依赖、补充新增配置字段并运行完整自检。

实际配置、数据库、媒体和第三方工具均由 `.gitignore` 排除，不会被快进更新覆盖。

## Docker/QNAP 更新

```bash
git pull --ff-only
./docker/build-and-start.sh
```

四个宿主机映射目录必须保持不变。更新前建议备份 `CONFIG_DIR`，重大数据库迁移前同时备份 `MEDIA_DIR` 的索引。

## 回滚

代码回滚：

```bash
git fetch --tags
git checkout <已验证标签>
```

Docker 重新构建后仍使用同一组持久化目录。若版本包含数据库迁移，必须结合对应版本迁移说明和数据库备份回滚，不得只回退代码而忽略数据结构。

Windows 完整恢复包和 Git Bundle 用于 GitHub、网络或工具下载不可用时的离线恢复。
