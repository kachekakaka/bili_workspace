# 备份、恢复与旧版本迁移

## 1. 当前目录边界

Windows / 源码运行：

```text
config/      配置、标签定义和运行环境设置
userdata/    SQLite、任务快照、删除历史、下载索引、任务日志、缓存和临时状态
downloads/   永久媒体文件
```

Docker 容器：

```text
/data/config
/data/userdata
/downloads
```

宿主机分别由 `CONFIG_DIR`、`USERDATA_DIR`、`MEDIA_DIR` 映射。

## 2. Docker 需要备份

完整恢复建议备份：

```text
CONFIG_DIR    配置、标签定义、管理员初始化信息和 Bilibili 凭据
USERDATA_DIR  SQLite、标签关系、删除历史、任务、观看进度、索引和日志
MEDIA_DIR     永久媒体文件
```

可以按需排除：

```text
USERDATA_DIR/cache  封面和兼容副本，可重建
USERDATA_DIR/tmp    下载、转码和设备导出临时状态
```

但若直接备份整个 `USERDATA_DIR`，恢复流程最简单。

## 3. Windows 需要备份

```text
config/
userdata/
downloads/
BBDown_portable/BBDown.data   Bilibili 会话，必须加密和限制访问
.env                          本机监听地址和端口
```

`config/runtime.env` 已位于 `config/` 中。`userdata/cache` 和 `userdata/tmp` 可不备份，但数据库、索引和删除历史必须保留。

无需备份：

```text
.runtime/
.venv/
.pytest_cache/
__pycache__/
```

## 4. SQLite 一致性备份

推荐先停止服务或容器再复制。在线备份 SQLite 时必须同时保留：

```text
bili_workspace.db
bili_workspace.db-wal
bili_workspace.db-shm
```

不能只复制主 `.db`。数据库中包含：

- 作品库和媒体文件索引；
- 分组、标签定义同步结果和标签关系；
- 任务快照和导出状态；
- 管理员、会话和观看进度；
- `deleted_media` 删除历史。

删除历史不作为作品出现在媒体库，但用于搜索时标记“已删除”。

## 5. Docker 恢复

1. 停止容器；
2. 恢复原 `CONFIG_DIR`、`USERDATA_DIR` 与 `MEDIA_DIR`；
3. 确认 PUID/PGID 对三个目录可读写；
4. 恢复敏感凭据时确认文件权限；
5. 运行 `./docker/build-and-start.sh`；
6. 检查管理员、作品数量、分组、标签、删除历史、任务和播放记录。

## 6. Windows 恢复

1. 停止网站；
2. 恢复 `config/`、`userdata/`、`downloads/` 和需要的 `BBDown.data`；
3. 从 GitHub 更新源码；
4. 运行 `start.bat`；
5. 运行 `verify.bat`；
6. 核对作品、标签、删除历史、任务和播放记录。

便携 Python、BBDown 和 FFmpeg 可由仓库内 `vendor/windows/*.pack` 重新解压，不应从旧机器复制 `.runtime/`。

## 7. 当前版本自动迁移

启动时会在目标文件不存在时安全迁移旧位置：

```text
config/bili_workspace.db
  → userdata/bili_workspace.db

config/bili_workspace.db-wal / -shm
  → userdata/ 对应文件

downloads/.bili_index.json
  → userdata/indexes/library.json

downloads/.bili_logs/
  → userdata/task_logs/
```

目标位置已有数据时不得覆盖。迁移后应保留旧备份，确认新版本可用后再清理。

V0.6 schema 升级会在 `userdata/backups/` 自动生成 `bili_workspace-v<旧版本>-YYYYMMDD-HHMMSS.db`，迁移事务和外键检查全部成功后只保留最近 3 份。升级失败时主数据库回滚到原状态并停止启动；回滚旧代码时必须先停服务，再恢复对应备份，不能只回退代码。高于当前程序支持版本的数据库会被拒绝启动，不会被静默降级。

## 8. 从旧 Windows 版本迁移

- 旧根目录 `config.json`：当 `config/config.json` 不存在时，配置同步会复制旧文件并补充新版缺失字段；
- 旧 `downloads/`：整体保留或复制到新 `downloads/`，旧索引会迁移到 `userdata/indexes/`；
- 旧数据库：复制到原来的 `config/bili_workspace.db` 或直接恢复到 `userdata/bili_workspace.db`；
- 不复制旧 `.venv` 或 `.runtime`，由当前仓库重新生成；
- 已登录的 `BBDown.data` 可保留，但不要提交 Git 或通过聊天传输。

## 9. 从 V0.4 Windows 迁移到 QNAP

1. 停止旧版；
2. 将旧 `downloads` 中的媒体内容复制到 Docker 的 `MEDIA_DIR`；
3. 将旧 `.bili_index.json` 放在媒体目录根部，首次启动会迁移到 `USERDATA_DIR/indexes/`；
4. 将旧配置和数据库放入对应的 `CONFIG_DIR` / `USERDATA_DIR`；
5. 配置并启动容器；
6. 核对作品数量、路径、文件大小、分组和观看记录；
7. 确认无误后再决定是否删除 Windows 副本。

## 10. 删除历史与重新下载

删除作品后，媒体记录和文件从作品库移除，但 `deleted_media` 会保留作品标识和删除时间。搜索默认屏蔽这些作品；关闭屏蔽后可看到“已删除”。

用户主动重新下载成功后，应用会清除对应删除记录并标记为“已下载”。因此备份数据库时必须保留删除历史表，不能只备份媒体文件。

## 11. 不要手工修改路径

- 下载索引保存安全删除所需相对路径；
- SQLite 负责作品库分页、标签、分组、删除历史、观看进度、管理员、任务和导出状态；
- 分组重命名默认只更新逻辑归属，不大规模移动媒体文件；
- 不要手工写入任意路径，不安全路径会被拒绝。
